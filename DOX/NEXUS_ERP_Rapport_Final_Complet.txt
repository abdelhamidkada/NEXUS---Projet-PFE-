I. VUE D'ENSEMBLE ET CONTEXTE DU PROJET
================================================================================

1.1 Contexte et problématique
──────────────────────────────
  NEXUS ERP est un Système d'Information des Ressources Humaines (SIRH)
  full-stack développé dans le cadre d'un PFE au sein de l'entreprise
  DyxIA. Il répond à une problématique réelle : remplacer les outils
  fragmentés (feuilles Excel, emails pour les congés, dossiers papier)
  par une plateforme web centralisée, sécurisée et intelligente.

  La plateforme couvre l'intégralité du cycle de vie d'un collaborateur :
  de son intégration (onboarding) jusqu'à son suivi quotidien de présence
  et à l'analyse prédictive du risque de départ (attrition).

2.2 Pattern architectural Backend
───────────────────────────────────
  Le backend suit le pattern en couches classique de Spring :

  REQUEST HTTP
      │
      ▼
  JwtAuthenticationFilter (Filtre de sécurité)
      │
      ▼
  @RestController (Couche présentation API)
      │
      ▼
  @Service (Couche métier — logique, validation, calculs)
      │
      ▼
  @Repository / JPA (Couche accès données — Hibernate)
      │
      ▼
  MySQL 8.0

2.3 Pattern architectural Frontend
────────────────────────────────────
  L'application React utilise un pattern SPA (Single Page Application)
  sans routeur React Router externe — le routing est géré nativement
  via l'historique du navigateur (window.history.pushState) dans App.jsx.

  Flux de données :
  - État global    : Zustand avec persistance localStorage
  - API calls      : Axios avec intercepteurs JWT automatiques
  - Composants     : Fonctionnels + Hooks (useState, useEffect, useRef)
  - Notifications  : Système de "toast" intégré dans App.jsx

4.1 Principe — Versioning Flyway
──────────────────────────────────
  Flyway gère l'évolution du schéma de façon séquentielle et irréversible.
  Chaque fichier SQL est nommé V{n}__{description}.sql et appliqué dans
  l'ordre lors du démarrage de l'application.

  Configuration (application.yml) :
    spring.flyway.enabled: true
    spring.flyway.baseline-on-migrate: true
    spring.flyway.validate-on-migrate: false
    spring.jpa.hibernate.ddl-auto: validate

V. SÉCURITÉ — JWT, RBAC, CHIFFREMENT
================================================================================

5.1 Authentification JWT Stateless
────────────────────────────────────
  NEXUS ERP utilise une authentification 100% stateless via JSON Web Tokens.
  Il n'existe aucune session serveur — chaque requête est auto-portante.

  Flux de connexion :
  ┌──────────┐     POST /api/auth/login         ┌─────────────────┐
  │  Client  │ ──{email, password}──────────────►│ AuthController  │
  └──────────┘                                  └───────┬─────────┘
                                                        │ DaoAuthenticationProvider
                                                        │ → UserDetailsService.loadByUsername()
                                                        │ → BCrypt.matches(password, hash)
                                                        │
                                                ┌───────▼─────────┐
                                                │   JwtService    │
                                                │ generateToken() │
                                                └───────┬─────────┘
  ┌──────────┐     {token: "eyJhbGci..."}        │
  │  Client  │◄────────────────────────────────────┘
  │ localStorage via Zustand persist             │
  └──────────┘

  Requête authentifiée :
  ┌──────────┐  Authorization: Bearer <token>   ┌─────────────────────────┐
  │  Client  │──────────────────────────────────►│ JwtAuthenticationFilter │
  └──────────┘                                  │ • extractUsername()     │
                                                │ • isTokenValid()        │
                                                │ • setAuthentication()   │
                                                └───────────┬─────────────┘
                                                            │ SecurityContextHolder
                                                            ▼
                                                  @RestController protégé

  Paramètres JWT :
  - Algorithme signature ......... HMAC-SHA256 (HS256)
  - Durée de vie token accès ..... 8 heures
  - Durée de vie refresh token ... 7 jours (rotation à chaque renouvellement)
  - Clé secrète .................. Base64(32 bytes) dans application.yml
  - Bibliothèque ................. JJWT 0.12.6

5.2 Refresh Tokens Rotatifs
────────────────────────────
  Un mécanisme de refresh tokens a été implémenté pour les sessions longues :
  - Stockés en BDD (table refresh_tokens)
  - UUID unique par session
  - Révocables en cas de vol (is_revoked)
  - Rotation automatique : chaque usage génère un nouveau refresh token
  - Endpoint dédié : POST /api/auth/refresh

5.3 Chiffrement AES-GCM-256 des données sensibles
───────────────────────────────────────────────────
  FICHIER : AttributeEncryptor.java (implements JPA AttributeConverter)

  Ce convertisseur est appliqué de manière transparente par JPA lors de
  chaque lecture/écriture en base de données.

  Algorithme :
  - Mode ................. AES/GCM/NoPadding (mode authentifié)
  - Taille clé ........... 256 bits
  - IV (nonce) ........... 12 octets aléatoires (SecureRandom) par chiffrement
  - Tag GCM .............. 128 bits (intégrité + authenticité garanties)
  - Format stockage ...... Base64(IV_12bytes || CipherText) → VARCHAR(512)

  Champs protégés :
  - rib (Relevé d'Identité Bancaire) — sur EmployeeProfile

  Usage dans l'entité :
    @Convert(converter = AttributeEncryptor.class)
    @Column(length = 512)
    private String rib;  // Totalement transparent pour le code métier

5.4 Hachage des mots de passe
──────────────────────────────
  - Algorithme .............. BCrypt
  - Cost factor ............. 12 (environ 300ms par hash — résistance brute force)
  - Configuration ........... new BCryptPasswordEncoder(12)
  - Stockage ................ password_hash VARCHAR(255) dans la table users

5.5 Contrôle d'accès par rôle (RBAC)
──────────────────────────────────────
  5 rôles natifs, définis en base de données (V1) et assignés aux utilisateurs :

  ┌──────────────┬───────────────────────────────────────────────────────────────┐
  │ Rôle         │ Droits et périmètre d'accès                                  │
  ├──────────────┼───────────────────────────────────────────────────────────────┤
  │ EMPLOYEE     │ Profil personnel, pointeuse, demandes de congés propres       │
  │ MANAGER      │ + Validation N+1 des congés de son équipe                    │
  │ HR_ADMIN     │ + CRUD complet profils, traitement RH, exports PDF            │
  │ IT_ADMIN     │ + Gestion des tickets support, monitoring système             │
  │ DIRECTION    │ + Accès analytics, ML attrition, KPIs décisionnels            │
  └──────────────┴───────────────────────────────────────────────────────────────┘

  Protection front-end :
  - Composant RoleProtectedRoute dans App.jsx

      restreints à leur espace personnel, quel que soit leur département.
    * Les widgets analytiques (IA) et système ne sont accessibles qu'aux rôles
      privilégiés (MANAGER, HR_ADMIN, IT_ADMIN, DIRECTION) selon leur département. avec double validation :

5.6 Protection path traversal
───────────────────────────────
  FileStorageService.java protège contre les attaques de type path traversal :
  - Rejet de tout nom de fichier contenant ".."
  - Vérification que le chemin résolu reste dans fileStorageLocation
  - Exemple rejeté : ../../../etc/passwd

6.1 Gestion des profils employés (Fiche 360°)
──────────────────────────────────────────────
  13 collaborateurs pré-configurés (E0001 → E0013) avec :
  - Données professionnelles : poste, département, matricule, hiérarchie
  - Données d'identité       : CIN, adresse, contact
  - Données contractuelles   : type contrat (CDI/CDD/Stage), dates, durée
  - Données bancaires        : RIB chiffré AES-256
  - Médias                   : photo URL, signature numérique

  Services disponibles :
  - CRUD complet (Create, Read, Update, Delete)
  - Recherche par matricule, email, département
  - JOIN FETCH optimisé (anti N+1)
  - Pagination et filtres

  Endpoints (EmployeeProfileController.java) :
  POST   /api/v1/hr/profiles          — Créer un profil
  GET    /api/v1/hr/profiles          — Lister tous les profils
  GET    /api/v1/hr/profiles/{id}     — Récupérer un profil par ID
  PUT    /api/v1/hr/profiles/{id}     — Mettre à jour un profil
  DELETE /api/v1/hr/profiles/{id}     — Supprimer un profil
  GET    /api/v1/hr/profiles/{id}/pdf — Télécharger la fiche PDF

6.2 Matrice des compétences
─────────────────────────────

7.1 Pointeuse numérique géolocalisée et Cycle de Paie
─────────────────────────────────────────────────────

  - Gestion stricte de la journée de travail (shift standard de 08:00 à 16:00).
  - Tolérance de départ de 5 minutes (pointage de sortie à partir de 15:55 traité
    comme journée "Normale", pointage de sortie avant 15:55 traité en "Absence injustifiée").
  - Calcul des heures supplémentaires (limitées entre 16:00 et 20:00 max).
  - Cycle de paie dynamique glissant du 16 du mois M-1 au 15 du mois M.

  BACKEND :
  - Service TimeCalculationService.java :
    * Calcul du cycle de paie en deux requêtes (in-memory aggregation).
    * Calcul des heures normales vs supplémentaires.
    * Détection des pointages incomplets et absences injustifiées.
    * Calcul des heures de nuit (tranche 21:00 → 06:00) et reverse-geocoding.

  Endpoint pointage :
  POST /api/v1/time-tracking?employeeId={id}   — Enregistrer un pointage
  GET  /api/v1/time-tracking/report/{id}        — Rapport journalier

7.3 Gestion des congés — Machine à états
──────────────────────────────────────────
  MODÈLE : LeaveRequest.java

  Types de congés :
  - PAID             → Congés Payés
  - SICK             → Maladie
  - UNPAID           → Sans Solde
  - MATERNITY_PATERNITY → Maternité / Paternité

  Machine à états stricte :
              [Employé soumet]
                     │
                     ▼
                 PENDING
                /       \
          MANAGER        MANAGER
          Valide          Refuse
            │                │
            ▼                ▼
       VALIDATED_N1       REJECTED
            │
       HR_ADMIN traite
            │
            ▼
       PROCESSED_HR

  Règles métier :
  - Seul l'employé peut créer une demande en PENDING
  - Seul un MANAGER peut passer de PENDING → VALIDATED_N1 / REJECTED
  - Seul HR_ADMIN peut passer VALIDATED_N1 → PROCESSED_HR
  - Toute transition invalide lève InvalidLeaveTransitionException

  Leave Accrual automatique :
  - Planificateur Spring (@Scheduled) calcule automatiquement l'accumulation
    des droits à congés selon le type de contrat et l'ancienneté

8.2 Modèle de Machine Learning — Prédiction d'Attrition
─────────────────────────────────────────────────────────
  BIBLIOTHÈQUE : Smile (Statistical Machine Intelligence and Learning Engine) v3.0.2
  ALGORITHME : Random Forest (forêt aléatoire — ensemble d'arbres de décision)

  SERVICE : AttritionPredictionService.java

  Données d'entraînement synthétiques (100 lignes) :
  - Features numériques :
    * age (int, 22-60 ans)
    * absencesCount (int, 0-30 jours)
    * overtimeHours (double, 0-80 heures)
    * performanceScore (int, 1-100)
  - Label binaire : attritionRisk (0 = faible, 1 = élevé)

  Corrélations encodées :
  - Hautes absences (>10) + faible performance (<40) → HIGH_RISK
  - Heures sup élevées (>50) → HIGH_RISK
  - Faible ancienneté + performance faible → HIGH_RISK
  - Profil stable (performance >70, absences <5) → LOW_RISK

  Score OOB (Out-Of-Bag) moyen : ~89% de précision

  Endpoint REST (public, pas d'auth requise) :
  GET /api/v1/analytics/attrition-risk
  Paramètres : ?age={int}&absencesCount={int}&overtimeHours={double}&performanceScore={int}
  Réponse : "HIGH_RISK" ou "LOW_RISK" (texte brut)

  Initialisation :
  - Le modèle est entraîné à chaque démarrage de l'application (@PostConstruct)
  - Logs Smile visibles au démarrage avec les scores OOB par arbre

10.1 Architecture du Coffre-Fort
──────────────────────────────────
  La plateforme implémente un coffre-fort numérique basé sur le système
  de fichiers local, avec ségrégation stricte par matricule employé.

  Structure physique :
  d:/Dev/NEXUS/storage/
  ├── E0001/
  │   ├── contrat_cdi.pdf
  │   └── fiche_paie_mai.pdf
  ├── E0002/
  │   └── diplome_ingenieur.pdf
  └── E0013/
      └── attestation_travail_E0013.pdf

  Principe de ségrégation :
  - Chaque employé a son propre répertoire nommé d'après son matricule
  - Un employé ne peut pas accéder aux fichiers d'un autre
  - La protection path traversal empêche "../" dans les noms de fichiers

  Configuration :
    nexus.storage.path: d:/Dev/NEXUS/storage    (application.yml)
  → Extensible vers AWS S3 ou GCS sans changer le code métier

10.2 FileStorageService.java
─────────────────────────────
  Méthodes principales :

  storeFile(MultipartFile file, String matricule) : String
  - Nettoie le nom de fichier (StringUtils.cleanPath)
  - Vérifie l'absence de ".." (protection traversal)
  - Crée le répertoire /storage/{matricule}/ si inexistant
  - Copie le fichier (REPLACE_EXISTING)
  - Retourne le chemin relatif "E0001/fichier.pdf"

  loadFileAsResource(String filePath) : Resource
  - Résout le chemin absolu
  - Vérifie que le chemin reste dans fileStorageLocation
  - Retourne UrlResource si le fichier existe et est lisible
  - Lève ResourceNotFoundException sinon

  Initialisation @PostConstruct :
  - Crée automatiquement le dossier racine au démarrage

10.3 Endpoints du Coffre-Fort (DocumentController.java)
─────────────────────────────────────────────────────────
  Routes restructurées par matricule (Phase 1 - 17 Juin 2026) :

  POST  /api/v1/documents/{matricule}/upload
    → Paramètre multipart "file" + "documentType" (CONTRACT/PAYSLIP/ID_CARD/DIPLOMA/OTHER)
    → Sauvegarde physique + enregistrement en BDD (HrDocument)
    → Retourne HrDocumentResponse (id, type, chemin, date, is_signed)

  GET   /api/v1/documents/{matricule}/download/{filename:.+}
    → Construit le chemin "E0001/fichier.pdf"
    → Détection MIME dynamique (Files.probeContentType)
    → Retourne le fichier en Content-Disposition: attachment

  GET   /api/v1/documents/{matricule}/export/attestation
    → Génère l'attestation de travail en temps réel via OpenPDF
    → Ne persiste aucun fichier (génération à la volée)
    → Retourne application/pdf

  GET   /api/v1/documents/{matricule}/list
    → Retourne la liste de tous les documents du collaborateur

10.4 Génération PDF — PdfExportService.java (OpenPDF 1.3.32)
──────────────────────────────────────────────────────────────
  Deux méthodes de génération disponibles :

  A) generateEmployeePdf(Employee employee)
     → Fiche collaborateur simple (profil 360°)
     → En-tête bleu NEXUS, données professionnelles, pied de page
     → Utilisée par /api/v1/employees/{id}/export-pdf

  B) generateAttestationPdf(EmployeeProfile profile)
     → Attestation de travail officielle (usage légal)
     → Structure complète du document :

     1. BLOC EMPLOYEUR (haut gauche)
        "DyxIA — Direction des Ressources Humaines"
        "NEXUS ERP Platform — contact.rh@dyxia.fr"

     2. DATE D'ÉMISSION (haut droit)
        "Fait à {location}, le {date_FR}"

     3. TITRE CENTRAL (18pt, bleu NEXUS)
        "ATTESTATION DE TRAVAIL"
        Sous-titre : "Document officiel émis par le Département RH"

     4. TABLEAU IDENTITÉ COLLABORATEUR (2 colonnes)
        Nom Complet | Matricule | Email | Poste | Département | Lieu

     5. TABLEAU CONTRAT (2 colonnes)
        Nature du contrat | Date d'embauche | Ancienneté calculée

     6. CORPS LÉGAL (texte continu)
        "La société DyxIA atteste que M./Mme {nom}, matricule {mat},
        est employé(e) en qualité de {poste} au sein du département
        {département} depuis le {date_embauche}..."

     7. BLOC SIGNATURE (aligné à droite)
        "Pour la Direction des Ressources Humaines"
        [Zone signature + cachet]
        "DRH / Responsable Paie"

     8. PIED DE PAGE LÉGAL (7pt, italique)
        "ATT-E0001-2026 — Confidentiel — Ne pas diffuser sans autorisation RH"

  → Référence unique générée : ATT-{matricule}-{année}
  → Utilisée par /api/v1/documents/{matricule}/export/attestation

  FROM maven:3.9-eclipse-temurin-17 AS builder
  WORKDIR /app
  COPY nexus-erp/pom.xml .
  COPY nexus-erp/src ./src
  RUN mvn clean package -DskipTests

  # Stage 2 — RUNTIME
  FROM eclipse-temurin:17-jre-alpine AS runtime
  WORKDIR /app
  COPY --from=builder /app/target/*.jar app.jar
  EXPOSE 8080
  ENTRYPOINT ["java", "-jar", "app.jar"]

  - Image de production légère (JRE Alpine, ~100 MB vs ~400 MB avec JDK)
  - Les outils de build (Maven, JDK) ne sont pas inclus dans l'image finale

     - Attestations de travail PDF professionnelles (OpenPDF)
     - QR Codes sécurisés pour kiosk physique (ZXing)
     - Contenu légal complet avec identité employer et référence unique

     - Coffre-fort préparé pour migration S3 (interface commune)
     - Machine à états congés stricte et formellement définie
     - Pattern service/repository découplé (testable)

  5. Interface utilisateur premium
     - Dashboard role-based adaptatif (5 rôles, 4 layouts distincts)
     - Composants modulaires avec animations subtiles
     - Grille TimecardGrid haute densité (standard entreprise)
